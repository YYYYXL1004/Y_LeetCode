# 蓝桥杯 5 天冲刺可视化指挥台设计

## 1. 背景

目标是把 [notes/蓝桥杯/5天冲刺40分备考方案.md](../../../notes/蓝桥杯/5天冲刺40分备考方案.md) 做成一个可直接打开的静态可视化页面，参考 [notes/天梯赛/刷题进度追踪.html](../../../notes/天梯赛/刷题进度追踪.html) 的追踪体验，但不照搬其内容模型。

用户明确要求页面同时具备两类能力：

- 作战看板：能快速看到 `40 分目标`、`5 天节奏`、`ROI 分层`、`赛场策略`、`考前清单`
- 进度追踪：能对每日题目和任务进行勾选、统计、保存、导入导出

最终页面应适合冲刺期每天反复打开，而不是只读一次的静态说明页。

## 2. 设计目标

### 2.1 核心目标

1. 在首屏给出明确的冲刺全貌，让用户一眼知道这 5 天该做什么、不该做什么。
2. 把 Markdown 方案里的“按天执行内容”转成可勾选、可折叠、可统计的执行面板。
3. 用本地持久化保存备考进度，保证页面刷新后状态不丢失。
4. 保留参考页“仪表盘 + 交互卡片”的使用感，但视觉方向改为更贴近备赛的冲刺看板。

### 2.2 非目标

1. 不做 Markdown 实时解析器，页面数据直接以内嵌 JS 结构维护。
2. 不接入后端、数据库、账号系统或跨设备同步。
3. 不做复杂题解管理、标签体系编辑器或自动从题库抓取题目。
4. 不在本次实现中改写原始 Markdown 备考方案内容。

## 3. 页面定位

页面采用“单页指挥台”为主体，吸收“Tab 切换 + 折叠分区”的交互方式。

最终信息结构分三层：

1. `全局层`：顶部仪表盘，负责展示总体目标和进度摘要。
2. `总览层`：中部 Tab 区，负责展示 5 天路线、ROI 分层、赛场策略、考前清单。
3. `执行层`：下部按 Day 折叠的执行区，负责具体题目和任务打卡。

这保证用户既能先看全局，又能直接下钻到当天执行内容，不需要在多个页面之间来回切换。

## 4. 信息架构

### 4.1 顶部仪表盘

首屏头部展示以下信息卡：

- `目标分数`：固定显示 `40 分左右`
- `冲刺周期`：`Day 1 - Day 5`
- `总体完成度`：按已完成任务数和题目数折算为百分比
- `已完成题目数`
- `已完成任务数`
- `主攻 / 可冲 / 暂放` 三类题型摘要

头部还应包含一句策略性副标题，强调“真题主线 + 专题补洞”的执行原则。

### 4.2 Tab 区

Tab 区至少包含以下四个视图：

1. `总览`
2. `每日执行`
3. `赛场策略`
4. `考前清单`

默认进入 `总览`。

#### 总览 Tab 内容

- `5 天路线总览`：展示 Day 1 到 Day 5 的主目标、主攻年份/任务、关键产出
- `ROI 分层区`：A 类稳分题、B 类高 ROI 中档题、C 类高风险题
- `主攻题型`：枚举、模拟、字符串、筛法、简单搜索、DP 等
- `暂时放弃`：LCA、线段树、DSU on tree 等
- `补洞原则`：何时切专题，何时重写原题

#### 每日执行 Tab 内容

- Day 1 到 Day 5 的折叠卡片列表
- 每个 Day 包含目标、优先题单、补洞池、复盘重点、时间分配
- 默认展开 Day 1，其余折叠

#### 赛场策略 Tab 内容

- 开场 10 分钟策略
- 前 90 分钟策略
- 中段止损策略
- 最后 20 分钟检查清单

#### 考前清单 Tab 内容

- 重点复习笔记链接
- 个人模板页提醒
- 最后结论摘要

### 4.3 每日执行卡片结构

每个 Day 卡片固定包含：

- `Day 标题`
- `今日目标`
- `优先题单`
- `补洞池`
- `复盘重点`
- `时间分配`
- `任务清单`

其中：

- `优先题单` 里的真题项可单独勾选
- `补洞池` 里的 LeetCode 题目可单独勾选
- `任务清单` 用于表示“限时做题 / 对照题解复盘 / 重写卡题 / 模拟”等动作是否完成

## 5. 数据模型

页面数据在 HTML 内以内嵌 JS 常量维护，不依赖外部 JSON 文件。

### 5.1 静态数据结构

需要拆成以下几个主要对象：

- `dashboardMeta`
  - 目标分数
  - 页面标题
  - 副标题
  - 颜色映射
- `overviewData`
  - 主攻题型
  - ROI 分层
  - 暂时放弃
  - 执行原则
- `dayPlans`
  - 每天的目标
  - 真题题单
  - 补洞池
  - 复盘重点
  - 时间分配
  - 任务清单
- `strategyData`
  - 赛场策略分段内容
- `checklistData`
  - 考前清单与链接

### 5.2 状态数据结构

页面运行时维护两类状态：

- `problemStatus`
  - 以题目标识作为 key
  - value 为布尔值，表示该题是否已完成
- `taskStatus`
  - 以任务标识作为 key
  - value 为布尔值，表示该任务是否已完成

关键规则：

1. 同一道题若在多个 Day 中重复出现，必须共享同一个 `problemStatus`，避免重复统计。
2. 任务项按 Day 独立记录，不跨日共享。
3. 总体完成度由 `problemStatus` 与 `taskStatus` 共同计算。

### 5.3 标识规则

- 真题使用稳定 id，例如 `2025-A-寻找质数`
- LeetCode 题使用稳定 id，例如 `LC-204-计数质数`
- 任务项使用 `dayX-task-name` 形式，例如 `day1-timed-practice`

## 6. 交互设计

### 6.1 必须支持的交互

1. 点击题目卡片切换完成状态。
2. 点击任务项切换完成状态。
3. Tab 切换不同信息视图。
4. Day 卡片折叠 / 展开。
5. 统计卡、进度条、分层计数实时联动更新。
6. 导出当前进度为 JSON。
7. 导入之前保存的 JSON 恢复状态。
8. 重置全部进度。

### 6.2 建议支持的交互

1. 在 `每日执行` 视图内增加 `只看未完成` 筛选。
2. 对高 ROI 中档题增加视觉强调标签。
3. 对 Day 卡片增加“今日建议先做什么”的路线提示条。

### 6.3 不做的交互

1. 不做拖拽排序。
2. 不做富文本编辑。
3. 不做多用户协作。
4. 不做复杂动画或图表库接入。

## 7. 视觉方向

整体视觉应与参考页区分开，不使用其紫色主基调。

### 7.1 视觉关键词

- 冲刺感
- 作战看板
- 高信息密度但可控
- 暖色强调 + 中性色承载

### 7.2 建议配色

- 主强调色：橙色 / 赤陶色
- 次强调色：深蓝灰 / 墨色
- 背景：米白、浅暖灰
- 状态色：
  - 已完成：绿色
  - 高 ROI：橙红色
  - 暂放：灰色

### 7.3 布局原则

1. 首屏要有“仪表盘”感。
2. 中段信息块之间层次分明，避免一屏全是同质卡片。
3. 每日执行区允许信息密度更高，但通过折叠减少压迫感。
4. 移动端下卡片改为单列布局，Tab 保持可点，统计卡自动换行。

## 8. 组件拆分建议

虽然最终是单文件 HTML，但内部代码结构仍应清晰分层。

建议拆成以下渲染函数：

- `renderHeader()`
- `renderStats()`
- `renderTabs()`
- `renderOverview()`
- `renderDayPlans()`
- `renderStrategy()`
- `renderChecklist()`
- `updateStats()`
- `toggleProblem()`
- `toggleTask()`
- `toggleDayCollapse()`
- `switchTab()`
- `exportProgress()`
- `importProgress()`
- `resetProgress()`

这样既能保持参考页的开发方式，也能避免脚本继续膨胀成难维护的大段代码。

## 9. 状态持久化与导入导出

### 9.1 本地持久化

使用 `localStorage` 保存进度。

建议的 key：

- `lanqiaoSprintProblemStatus`
- `lanqiaoSprintTaskStatus`
- `lanqiaoSprintActiveTab`

### 9.2 导出格式

导出 JSON 至少包含：

- 导出时间
- `problemStatus`
- `taskStatus`
- 统计摘要

### 9.3 导入行为

导入时校验 JSON 结构：

- 若缺少 `problemStatus` / `taskStatus`，提示格式错误
- 若存在未知字段，忽略即可
- 导入成功后立刻刷新页面统计和显示状态

## 10. 错误处理与健壮性

页面需要处理以下情况：

1. `localStorage` 不可用时，页面仍可浏览，但提示“进度不会持久保存”。
2. 导入非法 JSON 时，要给出明确错误提示，不得静默失败。
3. 若某个题目在数据源中重复定义，应在初始化阶段去重，避免统计异常。
4. 若导入数据中包含旧版本字段，应尽量兼容，不因为次要结构差异导致整个导入失败。

## 11. 可访问性与可用性

1. 交互控件要有明显 hover / active 状态。
2. 卡片文字颜色对比度要足够，避免橙色背景上堆过浅文字。
3. 按钮与标签的文本不能只靠颜色表达状态，完成状态需有图标或文字辅助。
4. 移动端点击区域不能过小。

## 12. 验证计划

本次实现完成后至少做以下手动验证：

1. 页面可直接通过本地文件打开。
2. `总览 / 每日执行 / 赛场策略 / 考前清单` 四个 Tab 切换正常。
3. Day 折叠与展开正常。
4. 点击题目卡片后状态即时更新，刷新页面后仍保留。
5. 点击任务项后统计同步更新。
6. 同一题目若在多个区域出现，状态保持一致。
7. 导出 JSON 成功，导入同一 JSON 后状态正确恢复。
8. 导入非法文件时给出错误提示。
9. 桌面端和移动端宽度下布局均可阅读。

## 13. 实现边界

本次实现产物限定为：

- 一个新的静态 HTML 页面文件
- 页面内嵌样式与脚本
- 不修改原 Markdown 方案的正文结构

允许在实现时参考 `notes/天梯赛/刷题进度追踪.html` 的组织方式与交互模式，但页面内容、视觉方向、数据结构和文案应针对蓝桥杯 5 天冲刺方案重新设计。

## 14. 成功标准

当以下条件同时满足时，视为完成：

1. 用户打开 HTML 后，能在首屏明确看懂 5 天冲刺主线。
2. 用户能在一个页面内完成“查看策略”和“打卡执行”两件事。
3. 统计信息会随着打卡同步更新。
4. 刷新页面不会丢失进度。
5. 页面视觉风格明显区别于参考页，但交互体验保持同等级别的可用性。
